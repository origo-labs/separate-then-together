How Separate-Then-Together Works

This empirically-validated strategy follows a multi-stage process to maximize exploration, synthesis,
and documentation in agent collaboration.

Phase 1: Separate (Divergence)

Agents work in **epistemic isolation**, meaning history filtering includes **only the current agent‚Äôs prior actions**, preventing mutual awareness. This strategy yields the highest initial semantic differentiation and cluster purity, encouraging unique, deep implementations without early influence from partners.

Phase 2: Collaborative (Convergence)

After a set number of independent turns, agents transition to operating with **unfiltered conversation history**. They can now reference, critique, and build upon partner contributions, allowing for synthesis and cross-domain integration. This phase helps resolve conflicts and consolidate diverse ideas into a coherent, actionable plan.

Phase 3: Report Generation (Synthesis)

Upon completion of the collaborative phase, a specialized Reporter Agent analyzes the entire session history. It synthesizes the evolved ideas, decisions, and conflicts into a comprehensive, structured document (e.g., a Technical Design Document or Implementation Plan). This automated reporting ensures that the valuable insights from the session are captured in a professional, shareable format.

Secure Financial API Architecture: A Separate-Then-Together Success Story

This case study demonstrates the effectiveness of the **Separate-Then-Together** multi-agent collaboration strategy. By initiating two specialized agents‚Äîa **Software Architect** and a **Security Engineer**‚Äîin isolation before bringing them together, the system produced a highly sophisticated, evolved architecture that balances scalability with rigorous security.

The Challenge

Topic: Design a secure, scalable REST API for a financial services platform.

Goals: Handle high transaction volumes (Scalability), Ensure data integrity and regulatory compliance (Security), Manage distributed transactions without data loss.

The Process

Phase 1: Separation (Turns 1-6)

The agents worked independently, unaware of each other.

  • The Architect focused on: Microservices using the Saga Pattern, Event-Driven Architecture with Kafka, Schema Management strategies.
  • The Security Engineer focused on: OAuth 2.0 and mTLS for authentication, Data Loss Prevention (DLP) and dynamic redaction, Policy Enforcement Modules.
Phase 2: Collaboration (Turns 7-17)

The agents were brought together. Their independent ideas collided and evolved.

  • Synthesis: The Architect integrated the Security Engineer's "Policy Module" into the event stream.
  • Conflict Resolution: They identified a core tension between speed (Architect) and compliance (Security).
  • Innovation: They co-created a novel abstraction: the **Distributed Conflict Resolution Engine (DCRE)**. This component didn't exist in either initial plan but emerged from the need to reconcile schema changes with security policies in real-time.

The Result: A Comprehensive Implementation Plan

The final output was not just a chat log, but a structured **Integrated Architecture Design Document** generated automatically by the system.

Key Architectural Decisions
  1. Distributed Conflict Resolution Engine (DCRE): Orchestrates conflicts during Saga lifecycle, consulting Schema Manager and Policy Enforcement Module.
  2. Dynamic Policy Enforcement: Assigns a **Risk Score** to transaction events, triggering stricter DCRE validation for high-risk events.
  3. Self-Healing Data Redaction: DLP engine detects sensitive data, DCRE applies redaction rules, preventing leaks even during failures.
  4. Human-in-the-Loop Governance (DGB): DCRE escalates unresolvable conflicts for human review, addressing regulatory nuances.

The "Separate-Then-Together" strategy proved superior to standard "chat" collaboration by avoiding groupthink, forcing innovation, and yielding high coherence in the final specification.

Quick Start

Get up and running with Separate-Then-Together in minutes.

Installation

# Clone the repository
git clone https://github.com/origo-labs/separate-then-together
cd separate-then-together

# Install with uv (recommended)
uv sync

# Or with pip
pip install -e .

Using Ollama (Local Models)

# Start Ollama and pull a model
ollama pull gemma3:4b

# Set environment variables
export OPENAI_API_KEY=ollama
export OPENAI_BASE_URL=http://localhost:11434/v1
export OPENAI_MODEL=gemma3:4b

# Run the CLI
separate-then-together --topic "Plan a microservices migration"

Using OpenAI

# Set your OpenAI API key
export OPENAI_API_KEY=sk-...
export OPENAI_MODEL=gpt-4o

# Run the CLI
separate-then-together --topic "Design a new authentication system"

Python API Example

from separate_then_together import (
    Config,
    LLMAgent,
    Persona,
    PersonaSelector,
    SessionEngine,
    SeparateTogetherStrategy,
)

# 1. Create configuration
config = Config.from_env()

# 2. Define personas
personas = [
    Persona(
        name="System Architect",
        system_prompt="You are a system architect focused on..."
    ),
    Persona(
        name="Security Engineer",
        system_prompt="You are a security engineer focused on..."
    ),
]

# 3. Select dissimilar pair
selector = PersonaSelector(personas, config.embedding_model)
persona1, persona2 = selector.select_dissimilar_pair()

# 4. Create agents
agent1 = LLMAgent(persona1, config)
agent2 = LLMAgent(persona2, config)

# 5. Create strategy
strategy = SeparateTogetherStrategy(
    separate_turns=10,
    collab_turns=20
)

# 6. Run session
session = SessionEngine(
    agent1, agent2,
    topic="Your planning task",
    strategy=strategy,
    config=config
)
results = session.run()

# 7. Export results
session.export_to_json("results.json")
session.export_to_markdown("results.md")

Research Background

Built on robust academic findings, ensuring impactful outcomes for your agentic workflows.

Persona Diversity for Novelty

Dissimilar persona pairs, quantified by low cosine similarity of their system prompt embeddings, consistently generate ideas with significantly higher **Novelty** and **Depth** scores. This promotes semantic variance and conceptual reach.

Separate-Then-Together: Optimal Hybrid

The **Separate-Then-Together** strategy is empirically validated as superior, producing the highest scores for both Novelty and Depth by effectively balancing initial exploration with subsequent synthesis.

Epistemic Isolation

Strict enforcement of **epistemic isolation** during the initial *Separate* phase‚Äîlimiting context to only an agent's own prior actions‚Äîis critical for maximizing the semantic diversity of initial ideation.

Cross-Domain Synthesis

The framework's effectiveness is rooted in enabling **cross-domain synthesis** during the Collaborative phase. This allows agents with different knowledge bases to integrate their distinct insights, leading to more elaborated and implementation-ready solutions.
